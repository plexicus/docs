# Scanner Catalog \{#reference_scanner-catalog-1\}

Plexicus orchestrates 25+ scanners across SAST, SCA, secrets, IaC, container, and cloud-posture domains. This page is the definitive list with default config, override syntax, and links to upstream docs.

Configuration entry points:

- **Per-client tool list** — `POST /client/included-tools/add` / `/remove`. Adds or removes a scanner from your org's default included set.
- **Per-tool parameters** — `POST /custom-parameters/{tool_name}` (full CRUD). Override default rule selection, severity thresholds, file patterns.
- **Per-tool config values** — `PUT /values/{tool_name}`. Provider-specific knobs (BlackDuck server URL, Checkmarx token).
- **Tool config YAML** (self-hosted) — `fastapi/security_tools_config.yaml`. Defines the canonical command-line, output format, parser, severity mapping for every tool.

## SAST — Static Application Security Testing \{#reference_scanner-catalog-2\}

Code-pattern scanners that find injection, deserialization, weak crypto, etc.

| Scanner | Languages | Default? | Notes |
|---|---|:-:|---|
| **Opengrep** | 30+ via Semgrep rules | ✅ | Plexicus's default SAST. Fork of Semgrep. |
| **Bandit** | Python | ✅ | Python-specific. Auto-skipped on non-Python repos. |
| **Semgrep** | 30+ | — | Same rule grammar as Opengrep; pick one or the other |
| **Checkmarx CxSAST** | Most languages | — | License required. Configure server URL via `/values/checkmarx`. |
| **Fortify SCA** | Most languages | — | License required. |
| **CodeQL** | JS/TS, Python, Java, C/C++, Go, Ruby, Swift | — | Add via `POST /client/included-tools/add` with name `codeql`. Free for OSS; paid for private. |

**Override examples:**

```bash
# Add a Bandit custom rule via the parameters API
curl -X POST "https://app.plexicus.ai/custom-parameters/bandit" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"rule_id": "B999", "severity": "High", "pattern": "subprocess\\.call\\(.*shell=True"}'

# Disable Bandit for the org
curl -X POST "https://app.plexicus.ai/client/included-tools/remove" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"tool": "bandit"}'
```

## Secrets — Credential Scanning \{#reference_scanner-catalog-4\}

| Scanner | Coverage | Default? | Notes |
|---|---|:-:|---|
| **Gitleaks** | Generic + 100+ provider patterns | ✅ | Default secrets scanner. Git-history aware. |
| **TruffleHog** | Generic + verified-secrets API checks | ✅ | More expensive but verifies secrets are still active |
| **Trivy Secrets** | Same as Gitleaks roughly | — | Use if Gitleaks/TruffleHog already cover; saves cost |

**Custom secret pattern:**

```bash
curl -X POST "https://app.plexicus.ai/custom-parameters/gitleaks" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"rule_id": "internal-token", "regex": "INTERNAL_[A-Z0-9]{32}", "tags": ["internal"]}'
```

## Output normalization \{#reference_scanner-catalog-10\}

Every scanner above emits a different format (JSON, SARIF, XML). The `tool-parser` service normalizes them into the unified Finding schema:

```text
Raw output → parser (per-tool) → standardizer → rule_enricher → CosmosOptimizer → MongoDB
```

The result: regardless of which scanner reported it, every finding has the same fields ([Findings Model](/docs/concepts/findings-model#concepts_findings-model-3)).

## Reading the active tool list \{#reference_scanner-catalog-11\}

`GET /client` returns your client document, including `included_tools`:

```json
{
  "included_tools": ["opengrep", "bandit", "trivy", "gitleaks", "checkov", "kics", "trivy_sca", "syft"],
  "excluded_tools": ["clamav"]
}
```

`GET /values` returns your custom tool config values:

```bash
curl -X GET "https://app.plexicus.ai/values" \
  -H "Authorization: Bearer $TOKEN"
```

## Adding a tool not in this catalog \{#reference_scanner-catalog-12\}

If you have a scanner Plexicus doesn't natively support (legacy Veracode, internal tooling), use the [Upload External Scan](/docs/recipes/upload-external-scan) recipe — base64 the SARIF output and POST to `/upload_scan/{repository_id}`. Plexicus's parser pipeline handles SARIF generically.

For native support of a new tool: open an issue at [github.com/plexicus/docs](https://github.com/plexicus/docs) with the tool name, output format, and a sample report.
