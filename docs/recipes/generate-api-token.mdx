# Generate an API Token \{#recipes_generate-api-token-1\}

API tokens authenticate programmatic calls to the Plexicus REST API — typically from CI/CD pipelines, scripts, or third-party integrations. They are bearer tokens scoped to a single user account and inherit that user's role.

:::warning Treat tokens like passwords
Anyone with the token can act as you. Store them in your secret manager, not in repository files. If a token leaks, revoke it immediately (last section of this page).
:::

## Generate \{#recipes_generate-api-token-2\}

<Steps>
  <Step title="Open Account → API Tokens">
    Navigate to **Settings → API Tokens** (or hit the page directly at [app.plexicus.ai/settings#api-tokens](https://app.plexicus.ai/settings#api-tokens)).
  </Step>
  <Step title="Click Generate Token">
    A dialog asks for two fields:
    - **Name** — short label so you can identify it later (e.g. *github-actions-prod*).
    - **Expires at** — optional ISO 8601 timestamp. Leave blank for non-expiring (not recommended).
  </Step>
  <Step title="Copy the token immediately">
    Plexicus calls `POST /user/generate-api-token` and shows the token **once**. After you close the dialog, the token is hashed in the database and **cannot be retrieved**. Store it in your secret manager now.
  </Step>
</Steps>

API equivalent:

```bash
curl -X POST "https://app.plexicus.ai/user/generate-api-token" \
  -H "Authorization: Bearer $YOUR_SESSION_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name": "github-actions-prod", "expires_at": "2026-12-31T23:59:59Z"}'
```

Response:

```json
{
  "success": true,
  "message": "API token generated successfully",
  "data": {
    "token": "plx_api_abcdef123456...",
    "name": "github-actions-prod",
    "expires_at": "2026-12-31T23:59:59Z"
  }
}
```

## Use the token \{#recipes_generate-api-token-3\}

Pass the token in the `Authorization` header:

```bash
curl -X GET "https://app.plexicus.ai/findings" \
  -H "Authorization: Bearer plx_api_abcdef123456..."
```

The token grants the same permissions as the user who created it. Tokens generated by an Admin can manage organization settings; tokens generated by a Developer can only see their own assigned findings.

## List your tokens \{#recipes_generate-api-token-4\}

`GET /user/api-tokens` returns a list of metadata for every token you've issued — never the token value itself.

```bash
curl -X GET "https://app.plexicus.ai/user/api-tokens" \
  -H "Authorization: Bearer $TOKEN"
```

```json
{
  "success": true,
  "data": [
    { "name": "github-actions-prod", "created_at": "2026-01-15", "expires_at": "2026-12-31", "token_type": "api" },
    { "name": "local-dev", "created_at": "2026-03-02", "expires_at": null, "token_type": "api" }
  ]
}
```

## Rotate a token \{#recipes_generate-api-token-5\}

There is no "rotate" endpoint. The pattern is **issue new + revoke old**:

<Steps>
  <Step title="Generate the replacement">
    Same flow as above. Use a new name (e.g. *github-actions-prod-2*) so the old one is still distinguishable.
  </Step>
  <Step title="Update the consumer">
    Push the new token to your secret manager and let your CI/CD pipeline pick it up.
  </Step>
  <Step title="Verify the new token works">
    Run a non-mutating call (`GET /session`) from the pipeline to confirm the new token authenticates.
  </Step>
  <Step title="Revoke the old token">
    See next section. Don't skip — orphaned tokens are how secrets stay leaked for months.
  </Step>
</Steps>

## Revoke a token \{#recipes_generate-api-token-6\}

In the UI: **Settings → API Tokens**, click the trash icon next to the token, confirm.

API:

```bash
curl -X DELETE "https://app.plexicus.ai/user/api-tokens/{token_name}" \
  -H "Authorization: Bearer $TOKEN"
```

Revocation is immediate. The next request using the revoked token returns `401 Unauthorized`.

## Token scope and limitations \{#recipes_generate-api-token-7\}

| Property | Value |
|---|---|
| Scope | Inherits user's role (Admin / Cyberoper / Developer) |
| Lifetime | Up to user; recommended ≤90 days |
| Rate limit | Same as session-based requests |
| Renewable? | No — generate a new one |
| 2FA bypass | Yes — tokens skip 2FA challenges |
| SSO bypass | Yes — tokens work even when SSO is required for UI |

The 2FA / SSO bypass is intentional for CI/CD use cases, but it's also a reason to keep token lifetimes short and rotate regularly.
